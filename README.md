# BioPet

생성 AI를 활용해 반려동물의 건강 데이터를 정리하고, 이상 징후 위험도를 분석한 뒤, 사람이 이해하기 쉬운 건강 리포트를 제공하는 팀 프로젝트입니다.

> 현재 단계: **주제 확정 완료 / 세부 기획 전 / MVP 및 문서 정의 필요**

---

## 1. 프로젝트를 한 줄로 설명하면

사용자가 반려동물의 기본 정보와 건강 관련 데이터를 입력하면,
시스템이 **건강 이상 위험도를 분석**하고,
그 결과를 바탕으로 **AI 설명 리포트**를 제공하는 웹 서비스입니다.

---

## 2. 왜 이 주제로 가는가

처음 주제는 `생성 AI 기반 반려동물 생체정보 분석을 통한 질병 예측 프로그램` 이었지만,
학생 팀 프로젝트에서는 범위가 너무 넓으면 중간에 무너질 가능성이 큽니다.

그래서 다음 방향으로 정리해서 진행합니다.

### 권장 프로젝트 방향
- **핵심 기능 1:** 반려동물 건강 데이터 입력
- **핵심 기능 2:** 이상 징후 위험도 분석
- **핵심 기능 3:** AI 건강 리포트 생성

### 이렇게 바꾸는 이유
- `질병 예측`은 데이터셋과 모델 난이도가 높음
- `생성 AI`는 예측 자체보다 **결과 설명**에 쓰는 것이 현실적임
- 비전공자 팀원도 역할을 나눠서 구현하기 쉬움

---

## 3. MVP(최소 기능) 범위

이번 프로젝트는 아래 5가지를 우선 구현하는 것을 목표로 합니다.

1. 반려동물 등록
2. 건강 데이터 입력
3. 위험도 분석 결과 확인
4. AI 건강 리포트 생성
5. 이전 분석 기록 조회

### 예시 입력값
- 이름
- 종(강아지/고양이)
- 나이
- 체중
- 활동량
- 식사량
- 수면시간
- 특이사항

### 예시 출력값
- 정상 / 주의 / 위험
- 위험 사유 요약
- 관리 팁
- 날짜별 분석 이력

## 4. 팀 역할 설명

중요한 점은 역할 이름보다 **무엇을 끝내야 하는지**가 더 중요하다는 것입니다.
각 역할은 아래처럼 이해하면 됩니다.

### 4-1. 정찬 — PM
**PM은 팀장이 아니라 진행 책임자입니다.**
직접 모든 개발을 다 하는 사람이 아니라, 팀이 멈추지 않게 만드는 역할입니다.

#### 해야 할 일
- 프로젝트 범위 정리
- 회의 진행 및 회의록 정리
- 요구사항 정의서 초안 작성
- 일정표 작성 및 마감 체크
- 화면/DB/API 문서가 서로 맞는지 확인
- 발표 흐름과 데모 시나리오 정리
- 이슈 발생 시 우선순위 결정

#### 결과물
- 요구사항 정의서
- WBS 또는 일정표
- 회의록
- 기능 우선순위표
- 발표 시나리오

#### PM이 꼭 기억할 것
- PM은 가장 잘하는 개발자가 아니라도 됨
- 대신 **정리, 확인, 기록, 일정 관리**를 해야 함
- 팀원이 무엇을 해야 할지 헷갈리지 않게 만드는 것이 가장 중요함

---

### 4-2. 양찬석 — Fullstack / API 연계
프론트와 백엔드를 연결하는 기준을 맞추는 역할입니다.

#### 해야 할 일
- API 명세서 구조 이해
- Swagger 문서 작성 또는 정리
- 프론트 요청 형식과 백엔드 응답 형식 통일
- 공통 응답 구조 설계
- 연동 과정에서 발생하는 오류 확인

#### 결과물
- API 명세서
- Swagger 문서
- 요청/응답 예시 JSON
- 프론트-백 연동 체크리스트

#### 쉽게 말하면
화면에서 입력한 데이터가 서버로 어떻게 가고,
서버 결과가 화면으로 어떻게 돌아오는지 관리하는 역할입니다.

---

### 4-3. 구승윤 — Frontend
사용자가 실제로 보게 되는 화면을 만드는 역할입니다.

#### 해야 할 일
- 화면 흐름 파악
- 페이지 구성
- 입력 폼 만들기
- 결과 화면 구성
- 버튼, 카드, 테이블 같은 UI 컴포넌트 구성
- API 연결 후 화면에 데이터 표시

#### 결과물
- 화면정의서 기반 UI
- 주요 페이지 구현
- 공통 컴포넌트
- 사용자 입력/결과 출력 화면

#### 쉽게 말하면
사용자가 클릭하고, 입력하고, 결과를 보는 화면을 만드는 역할입니다.

---

### 4-4. 황두경 — Backend
서버 로직을 만드는 역할입니다.

#### 해야 할 일
- FastAPI 기본 구조 이해
- API 엔드포인트 작성
- 입력값 검증
- 서비스 로직 구현
- DB와 연결
- AI 결과를 받아서 응답으로 전달

#### 결과물
- FastAPI 서버
- CRUD API
- 분석 요청 API
- 리포트 생성 API

#### 쉽게 말하면
화면에서 들어온 요청을 받아서 처리하고,
필요한 데이터를 DB나 AI 모델과 연결해서 결과를 반환하는 역할입니다.

---

### 4-5. 서병덕 — Data / AI
데이터와 모델을 담당하는 역할입니다.

#### 해야 할 일
- 사용할 데이터셋 조사
- 컬럼 의미 파악
- 결측치/이상치 처리
- 전처리 과정 정리
- 간단한 베이스라인 모델 생성
- 결과 해석 자료 정리

#### 결과물
- 데이터셋 정리표
- 전처리 규칙 문서
- 모델 학습 결과
- 성능 평가표

#### 쉽게 말하면
어떤 데이터를 써서 어떤 기준으로 예측할지 정하고,
실제로 분석 모델을 만드는 역할입니다.

---

### 4-6. 홍은수 — Database / Server
데이터를 저장하는 구조를 설계하는 역할입니다.

#### 해야 할 일
- ERD 작성
- 테이블 설계
- 정규화 검토
- PostgreSQL 연결
- 기본 CRUD 동작 확인
- 서버 실행 환경 보조

#### 결과물
- ERD
- 테이블 정의서
- DB 스키마
- 초기 SQL 또는 ORM 모델

#### 쉽게 말하면
반려동물 정보, 입력 기록, 분석 결과를 어떤 형태로 저장할지 설계하는 역할입니다.

## 6. 브레인스토밍 회의에서 꼭 결정해야 할 것

다음 주 회의 목표는 "아이디어 이야기"가 아니라 **결정**입니다.

회의가 끝났을 때 아래가 정해져 있어야 합니다.

1. 대상 동물: 강아지만 할지 / 고양이까지 할지
2. 입력 데이터 항목: 어떤 값을 받을지
3. 출력 결과: 위험도인지, 질병명인지
4. MVP 기능: 무엇까지 만들지
5. 화면 개수: 몇 개의 페이지가 필요한지
6. DB에 저장할 정보: 어떤 테이블이 필요한지
7. AI 사용 위치: 예측에 쓸지, 리포트에 쓸지

3. 건강 데이터 입력
4. 분석 요청
5. 결과 리포트 확인
6. 이전 기록 조회

### 3단계. 기능 후보 나열 (20분)
- 필요한 기능을 전부 적기
- 필수 / 선택 기능 구분

### 4단계. MVP 확정 (20분)
- 이번 프로젝트에서 꼭 필요한 기능만 남기기
- 어려운 기능은 후순위로 빼기

### 5단계. 역할별 산출물 나누기 (20분)
- 누가 무엇을 언제까지 할지 정하기
- 조사만 하지 말고 결과물을 정하기

### 6단계. 다음 회의 전 할 일 정리 (10분)
- 각자 과제 부여
- 마감일 정하기

## 12. PM이 흔들리지 않기 위한 기준

정찬이 전공자라고 해서 혼자 다 만들려고 하면 바로 무너집니다.
이번 프로젝트에서 PM은 아래 기준으로 움직이면 됩니다.

### 해야 하는 것
- 어려운 말을 쉬운 말로 바꿔서 설명하기
- 팀원들이 무엇을 해야 하는지 구체적으로 말해주기
- 작업 결과물을 문서와 마감일로 관리하기
- 모르는 것은 바로 정리해서 질문하기
- 기능 욕심 줄이고 끝낼 수 있는 범위로 자르기

### 하지 말아야 하는 것
- 혼자 다 구현하려고 하기
- 역할만 나누고 마감 기준을 안 정하기
- 회의에서 결론 없이 끝내기
- 기능을 계속 추가하기
- AI를 억지로 복잡하게 붙이기
